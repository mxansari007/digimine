## 5. Known issues observed

### Security
1. **Privilege escalation via unrestricted users create**: `allow create: if isOwner(userId)` with no field restrictions (`firebase/firestore.rules:226`, verified) while update blocks 'role' (rules:230-233). A fresh account with no users doc can client-setDoc role:'admin'/'super_admin'/'teacher'; `isDatabaseAdmin` reads that same field (rules:68-72). The register page legitimately depends on this hole (client writes role:'customer' at register/page.tsx:131,139); the server-only role-select route exists precisely because client role writes were meant to be blocked (api/auth/role-select/route.ts:5-13).
2. **Privileged overwrite via institute content endpoint**: `/api/institute/[instituteId]/content` trusts raw `body.slug` (route.ts:111) and for quizzes uses it directly as doc ID with admin-SDK `ref.set()` full replace (route.ts:112,158) — any institute admin can clobber an arbitrary `quizzes/{id}` doc including platform-published ones; inconsistently, other types get auto-IDs.
3. **testPurchases self-grant**: create allows any authenticated user with own userId and unconstrained status (`firestore.rules:642`, verified); a user writing `testPurchases/{uid_seriesId}` with status:'active' passes the paid-access check at rules:481-487 (verified).
4. **orders public read** `allow read: if true` (`firestore.rules:438`, verified) exposes order PII to anyone.
5. **jobs create unrestricted**: `allow create: if isAuthenticated()` (`firestore.rules:1015`, verified).
6. **Hardcoded bootstrap-admin god-mode emails** in rules (firestore.rules:58-65, verified — personal gmails) AND a separate hardcoded ADMIN_EMAILS list in admin-app client code (`apps/admin/src/contexts/AdminAuthContext.tsx:17-26`) which fabricates an admin AppUser client-side when no Firestore doc exists (:93-99). Three sources of admin truth overall: users.role, the `admins` collection (5 refs), and inline checks in `lib/server/adminListRoute.ts` vs `requireAdmin.ts`.
7. **BYPASS_PAYMENT_VERIFICATION** env flag disables payment verification when 1 (`apps/web/src/app/api/teacher/webhook/payment/route.ts:7-12`; key present in apps/web/.env.local) — dangerous if it leaks to production.
8. **Email verification is client-only**: no API route checks `decoded.email_verified` — /api/teacher/onboard (route.ts:8), /api/institute/register, /api/auth/role-select all run for unverified accounts (each only calls getBearerUserId, classroomAccess.ts:51-57). 10. **Dev bypasses**: usePhoneOtp accepts ANY 6-digit OTP on localhost or `NEXT_PUBLIC_OTP_DEV_BYPASS=1` (usePhoneOtp.ts:53-63) — env misconfiguration on a public deployment disables phone verification; EmailVerificationGate disables itself on localhost (EmailVerificationGate.tsx:78-86).
